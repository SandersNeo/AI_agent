# Работа агента с файлами

Документ описывает текущую реализацию файлового контура ИИ-агента: загрузку, хранение, извлечение текста, нативную передачу в модель и доступные инструменты.

## Что поддерживается

Сейчас агент умеет работать со следующими типами вложений:

- `txt`, `md`, `json`, `xml`, `log`, `bsl`
- `csv`, `tsv`
- `mxl`, `xls`, `xlsx`
- `doc`, `docx`
- `png`, `jpg`, `jpeg`, `bmp`, `gif`, `webp`
- `pdf`

## Маршрут файла

Текущий маршрут загрузки такой:

1. Пользователь нажимает `Прикрепить файл` в форме агента.
2. Клиент помещает файл во временное хранилище.
3. Сервер получает бинарные данные из временного хранилища.
4. Сервер сохраняет временную копию файла и запускает ingestion.
5. В диалоге сохраняются метаданные вложения, извлеченный текст, summary и, для `image/pdf`, исходный бинарник для нативной повторной передачи в модель.

Сервер не читает клиентский путь к файлу напрямую.

## Как обрабатываются типы файлов

### Текстовые файлы

`txt/md/json/xml/log/bsl` читаются локально через `ТекстовыйДокумент`.

Результат:

- полный `extracted_text` сохраняется в данных диалога
- в prompt попадает только ограниченный фрагмент

### Таблицы

`csv/tsv` читаются как текст с построением preview.

`mxl/xls/xlsx` читаются через `ТабличныйДокумент`.

Результат:

- сохраняется полный распознанный текст таблицы
- дополнительно строится `summary` с preview

### Word

`doc/docx` сейчас расшифровываются локально через `ТабличныйДокумент`, с fallback на обычное текстовое чтение.

Результат:

- файл классифицируется как `text`
- сохраняется `extracted_text`
- в модель уходит текстовый контекст, а не нативный бинарный input

### Изображения

`png/jpg/jpeg/bmp/gif/webp`:

- сохраняются как вложения типа `image`
- исходный бинарник сохраняется внутри данных диалога
- при вызове совместимой модели подготавливаются как нативный multimodal `image_url`

### PDF

`pdf`:

- сохраняется как вложение типа `pdf`
- исходный бинарник сохраняется внутри данных диалога
- при вызове совместимой модели подготавливается как нативный `file`

## Что хранится в диалоге

В данных диалога по вложению сохраняются:

- `id`
- `added_at`
- `file_name`
- `file_path`
- `extension`
- `kind`
- `mime_type`
- `size_bytes`
- `summary`
- `extracted_text`
- `native_transport`

Для `image/pdf` дополнительно сохраняется исходный бинарник во внутреннем поле `native_binary`.

Важно:

- полный `extracted_text` хранится целиком
- в prompt попадает только ограниченный фрагмент
- наружу через клиентские API бинарник не отдается

## Как вложения попадают в prompt

Для обычной работы в prompt добавляется блок `КОНТЕКСТ ВЛОЖЕНИЙ`.

Туда попадают:

- имя файла
- тип
- `summary`
- ограниченный фрагмент `extracted_text`

Это сделано, чтобы не раздувать prompt большими файлами.

## Нативная передача в модель

Сейчас нативно подготавливаются:

- изображения как `image_url`
- PDF как `file`

Подготовка делается на этапе сборки запроса к модели. Внутренний helper преобразует последнее пользовательское сообщение в multimodal `content`.

Это не отменяет текстовый attachment-context: summary и извлеченный текст по-прежнему доступны агенту и его инструментам.

## Инструменты модели для вложений

Для текстового режима агенту доступны встроенные инструменты:

- `get_attachment_text`
- `search_attachments`

Они позволяют:

- читать полный или частичный распознанный текст вложения
- искать строки по распознанным данным без раздувания основного prompt

Инструменты исполняются локально, а их результат возвращается модели через внутренний tool-loop.

## Логирование

В лог диалога пишутся события:

- добавление вложения
- запрос инструмента вложения моделью
- исполнение инструмента
- результат инструмента

Это позволяет видеть, когда агент реально читал вложение или искал по нему.

## Тесты

Файловый контур покрыт бесплатными тестами:

- текстовые вложения
- изображения
- `mxl`
- `xlsx`
- хранение полного `extracted_text`
- ограничение prompt по вложениям
- поиск по вложениям
- контракт и исполнение инструментов вложений
- нативная подготовка `image/pdf`
- локальная обработка `doc/docx`
- XDTO-безопасность сообщений

Запуск:

```bash
python automation/ops/run_tests.py
```

## Текущие ограничения

Пока еще не реализовано:

- OCR/vision-анализ изображения по пикселям на стороне модели с возвратом структурированного результата в UI
- полноценная локальная расшифровка сложных `doc/docx` с сохранением структуры документа
- структурированное представление таблиц по листам/строкам/колонкам
- UI списка вложений с просмотром summary и результатов поиска
- цитирование фрагментов вложения с точными позициями
